3️⃣ Parameter Matrix

Parameter Letter Typical Interpretation Key DevOps Focus
A – Availability A Expected uptime / RPO‑RTO. Redundancy, health‑checks, auto‑recovery, disaster‑recovery drills, chaos‑engineering validation.
B – Bandwidth / Performance B Throughput, latency, load spikes. Autoscaling (HPA, VPA, Cluster Autoscaler), multi‑layer caching, load‑balancing, performance‑testing in CI (k6).
C – Compliance & Security C Regulatory & audit mandates (PCI, GDPR, HIPAA). Secrets‑management (Vault, KMS), encryption‑at‑rest/in‑transit, OPA policy enforcement, CIS‑K8s hardening, audit logging.
D – Deployment & Automation D Delivery strategy (CI/CD, blue/green, canary). Immutable infrastructure, pipeline gating, automated rollback, feature‑flags, GitOps (ArgoCD/Flux).
E – Efficiency (optional) E Cost, resource utilization, carbon footprint. Rightsizing, spot‑instance usage, cost‑allocation tags, sustainability dashboards.

Note: The “E” column is optional but recommended for cloud‑native cost‑optimisation programs.

4️⃣ Extracting Parameters from a Request

  1. Identify the Service Goal – Ask the product owner: “What business outcome does this service enable?”
  2. Map Goal → Parameter Bucket – Example mappings:
  3. “99.9 % uptime SLA” → A
  4. “Peak 10 k RPS, < 100 ms latency” → B
  5. “Must be PCI‑DSS compliant” → C
  6. “Deploy via canary every 2 weeks” → D
  7. “Stay under $0.10 per request” → E (if used)
  8. Clarify Ambiguities – Prompt with targeted questions:
  9. “What is the acceptable RTO for a regional outage?”
  10. “Do you have any data‑residency constraints?”
  11. “Which environments (dev, staging, prod) need separate SLAs?”
  12. Document the Extraction – Populate a parameters.yaml (see Section 5) and record the rationale in the ticket’s Architecture Decision Record (ADR).

5️⃣ Canonical parameters.yaml & JSON‑Schema

5.1 parameters.yaml – Example (copy‑paste ready)

# parameters.yaml – source of truth for the service “order‑processor”
service:
  name: order-processor
  version: "2.4.1"

availability:
  targetUptime: "99.95%"          # internal SLO – external SLA may differ
  maxRTO: "30m"                  # Recovery Time Objective
  healthCheck:
    path: /healthz
    interval: 15s
    timeout: 5s
    failureThreshold: 3

performance:
  maxRPS: 5000
  latencyP99: "120ms"
  autoscaling:
    hpa:
      minReplicas: 3
      maxReplicas: 30
      cpuUtilizationTarget: 70
    vpa:
      enabled: true
  caching:
    enabled: true
    ttl: "5m"

compliance:
  standards:
    - PCI-DSS
    - GDPR
  dataClassification: "PII"
  encryption:
    atRest: true
    inTransit: true
    keyRotationDays: 90
  auditLogging: true
  policyEnforcement:
    opaGatekeeper: true
    conftest: true

deployment:
  strategy: canary
  canary:
    steps:
      - weight: 10
        pause: "5m"
      - weight: 30
        pause: "10m"
      - weight: 100
    successCriteria:
      - metric: latencyP99
        max: "150ms"
      - metric: errorRate
        max: "0.1%"
  rollback:
    onFailure: true
    maxRetries: 2
  gitOps:
    repo: git@github.com:your-org/order-processor.git
    path: helm/
    targetBranch: main

efficiency:                     # optional – omit if not used
  costTargetPerMRequests: 0.08   # USD
  spotInstanceAllowed: true
  carbonFootprintGoalKgCO2e: 1500
